8 - Azure availability zones and paired regions

1 - Availability zones

Availability zones - AZ are physical separated data-centers with an azure region, each availability zone is made up of one or more data-centers equipped with independent power, cooling, and networking.

You don't have a control to which data-center are your services deploy to, thats why Availability zones were created. You can select a specific availability zones by number AZ1, AZ2 or AZ3.

An availability zone is set up to be an isolation boundary from each other (so different buildings). If one zone goes down, the other continues working.

Availability zones are connected through high-speed, private fiber-optic networks, so they will be close enough to provide low-latency.

A region will contain at least 3 Availability Zones or more, if region have less than 3, then our region will not support availability zones feature.

Region that doesn't support availability zones is know as Alternative or Other regions. When deploying a high available mission-critical apps, its good idea to use regions that support available zones

Its common practice to run workload in at least 3 AZs to ensure services remain available in case 1 or 2 data-centers fails (High Availability).

Not all Azure Regions currently support availability zones. AZ costs more but it allows you to make applications highly available through redundancy, by replicating your compute, storage, networking, and data resources

Availability zones are primarily for VMs, managed disks, load balancers, and SQL databases. Each service that support availability zones fall into three categories.

A - Zonal services

Here you pin your resource to a specific region, resources/services like VMs, managed disks, IP addresses. With zonal services you can choose to which availability zone to deploy your service.

VMs are highly available environments, You can create a virtual machine and choose which availability zone that you want to deploy into.

# In azure portal, creating a new virtual machine and select our availability zone :-

For example, you can create two virtual machines for highly available, and specify that first VM goes to AZ 1, and second one goes to AZ 2 or 3.

Doing that will ensure that if case there's a data-center level failure, your virtual machine at least one them will continue working. Before without availability zones your virtual machines in a bog regions could be deployed to the same data-center or even to the same physical server.

B - Zone-redundant services

Azure platform replicates automatically across zones, this will automatically replicate your data a cross multiple availability zones. Zone-redundant service is enabled with zone-redundant storage, SQL Database services.

C - Non-regional/Always-available services

Services are always available across all Azure geographies and are resilient to zone-wide outages and region-wide outages. Non-regional is defined as where there is no dependency on a specific Azure region.
