Azure Resources, Resource Groups & Resource Manager

1 - Azure Resource

Resources are like an object that are used to represent a service in azure. All services are represented as generic object as Resources, here some services within azure.

  • Virtual machines (VMs)
  • Storage accounts
  • Web apps
  • databases
  • Virtual networks
  • Application Gateways
  • CosmosDB instances
  • App service
  • Cognitive services

# Microsoft Azure Services/Resources with their symbols :-

The moment you purchase a service in azure you create a resource with it. Anything you create, provision, deploy, etc. is a resource, when you done with the service and you delete it, the resource also deleted.

Resources are also your contracts which represent a service life-cycle.

Resource is an object saved as JSON format, it is used to store and manage a service in azure. Azure use these resources to save the configurations (options) that you make for your services.

These configurations are represented as a property on that resource, all resources in azure can be represented as JSON template.

# There is four common properties a cross all resource, here a json file with these properties :-

 {
  "type": "Microsoft.storage",
  "API_version": "2020-05-01",
  "name": "My_Storage",
  "locations": "US-West",
  ......
 }

Every resource has a different non-common properties that describe different services with azure.

In order to create a resource, we need to create a resource group (RG) first, no resources in azure can be created without a resource group.

2 - Resource Groups

Resource groups is a logical container for only resources, it's simply used for grouping related resources. When you create a resource, you're required to place it into a resource group.

Resource group is utility that provides a convenient way to group resources together to manage and organize them in a better way. You decide which resources belong in a resource group based on what makes the most sense for your organization.

Resource group were created as a utility to manage other resources

- More about group resources :-

  • Each resource can exist in only one resource group.
  • You can add or remove a resource to a resource group at any time.
  • When you create a resource group, you need to provide a location for that resource group.
  • The resources in a resource group can be located in different regions than the resource group.
  • You can move a resource (if it supports move operation) from one resource group to another group, but when you move a resource to a new group, it will no longer be associated with the former group.
  • Resource groups can't be nested, meaning you can't put resource group B inside of resource group A.

A resource group has its own meta-data with certain attributes, when you create a resource group and pick that region, the resource group lives and its information data stored in that particular region. (read more)

If a resource group's region is temporarily unavailable, you can't update resources in the resource group because the metadata is unavailable. The resources in other regions will still function as expected, but you can't update them. This condition doesn't apply to global resources like Azure Content Delivery Network, Azure DNS, Azure DNS Private Zones, Azure Traffic Manager, and Azure Front Door.

Q: Why we need to specify a location when we create a resource group ?

Because, resources in a RG can be a part of any region, but RGs are tied to a specific region. Its about meta data of the resources, which has to be stored in a location.

Inside the resource group you can have resources from many different regions in the same resource group. In resource group, you can have things mixed from multiple regions and resource types.

- Resource Group Naming Convention

Its good to have a consistent naming convention for your resource group, when deciding on a naming convention to standardize on, there are several different naming components to keep in mind.

- a consistent good naming convention for resource group for example :-

cloudarchitecture-prod-infrastructure-rg

  • what it's used for (cloudarchitecture)
  • environment (prod)
  • the types of resources contained within (infrastructure)
  • type of resource it is itself (rg = resource group)
  • another option is <Resource>-<subscriptionType>-<Region> like VNET-Shared01-WestUS

- Here are the most common naming convention components :-

  • Azure Region: The Azure Region where the resource is deployed.Such as East US 2 or Central US. An example shorter abbreviation for these might be E2 or C1.
  • Environment: The application lifecycle for the workload the resource belongs to; such as Development, UAT, Test, or Production. An example shorter abbreviation of these might be DEV or PRD.
  • Organization: An abbreviation or moniker representing your organization.
  • Workload or Application Name: The name of the application or workload the resource belongs to. Such as Jumpbox, Data Lake, AKS.
  • Business Unit: The top-level department or business unit of your company that owns or is responsible for the resource. Such as HR, Marketing, IT.
  • Instance: A number or letter indicating uniqueness when you have multiple instances of this resources for the same workload. Such as 01.
  • Resource Type: An abbreviation for the Azure Resource Type of this resource. Such as rg, vm, sql, aks.

- Resource Group Life Cycle

If you apply an action to a resource group, that action will apply to all the resources within the resource group. If you delete a resource group, all the resources will be deleted. Also if you grant or deny access to a resource group, you've granted or denied access to all the resources within the resource group.

- Resource Group Authorization

Resource groups are also a scope for applying role-based access control (RBAC) permissions. By applying RBAC permissions to a resource group, you can ease administration and limit access to allow only what's needed.
