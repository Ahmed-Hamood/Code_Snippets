1. What is Container ?

Container is a virtualization runtime environment for running apps, you can run multiple containers isolated and secured on a single physical or virtual host.

A container is a standard unit of software that packages up code and all its dependencies so the application runs quickly and reliably from one computing environment to another.

Container Bundles apps within an operating system of a single physical server using container environment runtime.

Container doesn't have an operating system it emulates operating system by using the host's operating system (VMs emulate hardware).

Container Much more lightweight and faster than VMs (seconds instead of minutes) because with container you don't manage the operating system. Containers they don't have their own operating system they use host's operating system

Container is a sandbox environment for each application, you can have multiple containers within a single runtime. Container run on top of a host operating system like VMs.

The major different in container is there is not operating system replicated on each container, the footprint of the container is much smaller that the one of virtual machine.

Containers bundle the libraries and components needed to run the application and use the existing host OS to run the container. You can run multiple isolated applications on each container. You can use same server or VM to host multiple container applications.

If you have five containers running on a server with a specific Linux kernel, all five containers and the apps within them share that same Linux kernel.

Containers are lightweight and designed to be created, scaled out, and stopped dynamically, they are designed for almost any scenario. It's possible to create and deploy virtual machines as application demand increases, but containers are a lighter weight, more agile method.

containers are designed to allow you to provide :-

  • Less Development efforts for your team
  • Less Maintenance, you don't need to maintain the operating system patches and updates
  • Less compute and storage requirement from these containers, because there is no operating system
  • Respond quicker to demand changes, so you can auto scale faster than with virtual machines

Containers are designed to allow you to respond quicker to changes on demand. With containers, you can quickly restart if there's a crash or hardware interruption. One of the most popular container engines is Docker, which is supported by Azure.

Containers are often used to create solutions by using a microservice architecture. This architecture is where you break solutions into smaller, independent pieces. For example, you might split a website into a container hosting your front end, another hosting your back end, and a third for storage or database. This split allows you to separate portions of your app into logical sections that can be maintained, scaled, or updated independently.

Why containers in your solutions ?

Imagine your website back-end has reached capacity but the front end and storage aren't being stressed. With containers, you could scale the back end separately to improve performance. If something necessitated such a change, you could also choose to change the storage service or modify the front end without impacting any of the other components.

2. Azure Container Instances (ACI)

Azure Container Instances are a platform as a service (PaaS) offering, the Simplest and fastest way to run a container immediately in Azure; without having or the need to manage any virtual machines or any additional services. You just upload containers and ACI will run them with automatic scaling.

Azure container instance is a serverless container because you don't need to manage the entire system like creating or configuring virtual machines or adopt to any additional services. Simply serverless containers mean run containers without managing servers.

In you company, you just ask your developers just to grab the application with configuration and other runtime middleware software that you need for this application to run, and bundle this application into a container image and hosted or upload it into container repository.

With container instances you don't need to run any servers zero nodes, you can have 20 container groups in a VM.

Azure Container Instances allow you to upload your containers into "container repository" and then the container instance service will run the containers for you.

We have a marketplace for public repositories of other images for containers, we can just grab an image and push it to container instance which will create a container for us.

Container instance will create a group of containers for you; underneath or behind the scene all containers are hosted inside a virtual machine.

You can host your containers like use one of them for web application that will exposed to users, and one for database storage that will be hidden from public/users.

3. What is Kubernetes ?

Kubernetes is a portable, extensible open-source platform for managing containerized workloads and services that facilitates both declarative configuration and automation. It makes sure that our services are running smoothly the way an app developer wants.

Kubernetes lets you run distributed system resiliently ,with scaling and failover for your applications.

Kubernetes is a container orchestrator that helps make sure that each container where is suppose to be, and that the container can work together. We can say its a conductor that managed everything.

Kubernetes can automatically remove failed containers, ot remove un-responsive ones, release an update and continuously check container health.

Kubernetes also handles storage, allowing admins to use varied storage types from local storage to cloud resources for containers data.

Also organizations can use Kubernetes to manage passwords, tokens, SSH keys amd other.

Kubernetes can also use DNS names or IP addresses to control how containers are made available. Also it has management capabilities like load balancer which distribute igh traffic volume a cross multiple container.

Kubernetes has features like rollout and rollbacks

Kubernetes can set and modify the preferred state for containers deployment, this allow admins to create new container instances in desired state, and then migrate existing containers to the new instances while removing the old ones.

administrator can tell Kubernetes which nodes are available and which resources are required for containers, and platform will automatically fit containers into those nodes to optimize resource use,

Kubernetes doesn't provide middleware or application services, we can deploy our services into containers managed by Kubernetes and accessed by other applications running thru Kubernetes.

4. Azure Kubernetes Service (AKS)

Azure Kubernetes Service is an open-source management platform, for managing and running a large number of containers by scheduling and running multiple containers.

AKS is highly scalable and customizable; it is designed for high scale and highly customizable deployment of containers. It is used to automate container deployment and manage containers life-cycle. AKS is the task of automating, managing, and interacting with a large number of containers.

As we know containers virtualize computer's resources abd provision those resources into instances that can run software services and applications. Containers might exist only for a short duration, which makes it impossible to create, start, organize, destroy and monitor containers manually.

Azure Kubernetes Service is a cloud-native available in any cloud provider like google cloud and AWS. Same as container principle you can either hosting with your own container images within container repository or using marketplace public repository.

As the deployment of container is finished Kubernetes exposes everything to users or applications thru a Load Balancer, it gives you the same ability like scales set either static scaling or auto scaling.

With AKS you can use those images and spread them across nodes (VMs underneath)), these nodes are separated and abstracted from you. Kubernetes is managing the deployment of containers a cross those nodes. We can have 3 nodes minimum and 100 nodes maximum.

Nodes is a set of IT resources as a physical or VMs that run one or more containers; A node supports a kubernetes Pod, which is a group of containers we can say Pod is one or more containers on a node, and pod is the smallest unit in kubernetes.

If node is removed then Kubernetes move affected workloads to different node. And if one pod crashes then Kubernetes creates new instance. Pods can be scaled manually or automatically (horizontal)

You can migrate existing apps to AKS :

  • Convert an existing application to one or more containers and then publish one or more container images to the Azure Container Registry.
  • Deploy the containers to an AKS cluster using the Azure portal or the command line.
  • Azure AD controls access to AKS resources.
  • You access SLA-backed Azure services, such as Azure Database for MySQL, via OSBA.
  • Optionally, AKS is deployed with a virtual network.

With Kubernetes you can :

  • You can manage storage, Persistent volumes to represent data storage to one or more containers. Data can be persisted across many pod instances. You can utilize cloud-based storage and data system e.g. Azure storage + Cosmos DB
  • You can manage networking, by exposing pods to internet, network isolation, Load balances traffic across multiple replicas of a pod.
    • Policy-driven network security.
    • Manage communication + name resolution between pods
  • It can be extended with additional capabilities, for example cloud events on pod creation, custom pod scheduling logic, on-demand provisioning of managed cloud services.

- Access, security, and monitoring

AKS lets you integrate with Azure AD to improve security and management by uses role-based access control that controls access and permissions to Kubernetes resources and namespaces to limit access to cluster resources.

You can configure an AKS cluster to integrate with Azure AD. With Azure AD integration, you can set up Kubernetes access based on existing identity and group membership. Your existing Azure AD users and groups can be provided with an integrated sign-on experience and access to AKS resources.

- Integrated logging and monitoring

Also AKS monitor the health of your cluster and resources. Azure monitor for Container Health collects memory and processor performance metrics from containers, nodes, and controllers within your AKS cluster and deployed applications. The container health log are stored

  • Stored in an Azure Log Analytics workspace.
  • Available through the Azure portal, Azure CLI, or a REST endpoint.

- Clusters and nodes

- Cluster node and pod scaling

AKS nodes run on Azure virtual machines (VMs). With AKS nodes, you can connect storage to nodes and pods, upgrade cluster components, and use GPUs.

A Kubernetes cluster is a set of nodes that run containerized applications. Containerizing applications packages an app with its dependences and some necessary services. They are more lightweight and flexible than virtual machines. In this way, Kubernetes clusters allow for applications to be more easily developed, moved and managed.

AKS supports Kubernetes clusters that run multiple node pools to support mixed operating systems and Windows Server containers.

As demand for resources change, the number of cluster nodes or pods that run your services automatically scales up or down. You can adjust both the horizontal pod auto-scaler or the cluster auto-scaler to adjust to demands and only run necessary resources.

- Cluster node upgrades

AKS offers multiple Kubernetes versions. As new versions become available in AKS, you can upgrade your cluster using the Azure portal or Azure CLI. During the upgrade process, nodes are carefully cordoned and drained to minimize disruption to running applications.

- GPU-enabled nodes

AKS supports the creation of GPU-enabled node pools. Azure currently provides single or multiple GPU-enabled VMs. GPU-enabled VMs are designed for compute-intensive, graphics-intensive, and visualization workloads.

- Confidential computing nodes (public preview)

AKS supports the creation of Intel SGX-based , confidential computing node pools (DCSv2 VMs). Confidential computing nodes allow containers to run in a hardware-based, trusted execution environment (enclaves) . Isolation between containers

- Mariner nodes

Mariner is an open-source Linux distribution created by Microsoft, and itâ€™s now available for preview as a container host on Azure Kubernetes Service (AKS).

The Mariner container host provides reliability and consistency from cloud to edge across the AKS, AKS-HCI, and Arc products. You can deploy Mariner node pools in a new cluster, add Mariner node pools to your existing Ubuntu clusters, or migrate your Ubuntu nodes to Mariner nodes.

- Storage volume support

To support application workloads, you can mount static or dynamic storage volumes for persistent data. Depending on the number of connected pods expected to share the storage volumes, you can use storage backed by either:

  • Azure Disks for single pod access, or
  • Azure Files for multiple, concurrent pod access.

- Virtual networks

An AKS cluster can be deployed into an existing virtual network. In this configuration, every pod in the cluster is assigned an IP address in the virtual network, and can directly communicate with:

  • Other pods in the cluster
  • Other nodes in the virtual network.

Pods can also connect to other services in a peered virtual network and to on-premises networks over ExpressRoute or site-to-site (S2S) VPN connections.

- HTTP application routing

The HTTP application routing add-on helps you easily access applications deployed to your AKS cluster. When enabled, the HTTP application routing solution configures an ingress controller in your AKS cluster.

5. Micro-services

Micro-services us an architecture solution where you break your app logic into smaller sections, independent pieces. This Allows you to separate portions of your app into logical sections that can be maintained, scaled, or updated independently.

Each service has small code-base that can be managed by a small development team. You can have different libraries or frameworks technologies for different teams, each team can choose the right tool for the job. Single development team can test & build & deploy a service.Each micro-service is completely autonomous with no cross-dependencies.

Provides fault isolation: If one goes down, it does not take out all application. Communicates with each other using APIs, APIs encapsulate internal functionality. Internal implementation details of each service are encapsulated behind their interface.

Micro-services is good for :

  • High release velocity
  • Highly scalable applications
  • Applications with rich domains/subdomains
  • Organizations with small development teams

- Micro-services deployment

  • Allows each microservice to be deployed independently of every other microservice.
  • A team can update an existing service without rebuilding/redeploying the entire application.
  • They can easily roll back & roll forward & update if something goes wrong.
  • Makes bug fixes + feature releases more manageable & less risky
  • Allows each microservice
    • to be scaled independently
    • persist own data & external state without common repository layer
  • For example, one for front-end, one for back-end and one for storage.
    • If back-end reaches capacity but not others, you can scale it individually.
    • Allows you replace storage microservice without affecting rest of the application.
