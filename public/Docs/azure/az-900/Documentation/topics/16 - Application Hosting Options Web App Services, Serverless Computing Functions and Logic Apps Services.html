1. Application Hosting Options

In order to host an applications then we probably need VM or container for the hosting solution. As we know VMs is the best hosting method and it gives you a full control over the hosting environment and allow to configure it exactly how you want.

Container is a compelling hosting solution option for isolating apps and individually manage each app separately with it own environment

What if i tell you that you can host your apps without the need of managing infrastructure like creating VMs and deploying Containers, Azure provides app services that will do all the hosting for us. You build your app with all packages and codes and send it to azure app service, and azure will host it for you.

- Azure App Service

Azure app service is a simple an HTTP-based service for hosting web applications (different runtimes and programming languages), it enables you to build and host web apps, background jobs, mobile back-ends, and RESTful APIs in the programming language of your choice without managing infrastructure.

Azure app service is PaaS offering, the main purpose of this service is building web applications. We just ask developers to prepare a simple deployment package/code and send it to app service.

App Service supports Windows and Linux. It enables automated deployments from GitHub, Azure DevOps, or any Git repo to support a continuous deployment model. Azure App Service lets you focus on building and maintaining your app, and Azure focuses on keeping the environment up and running.

App Service handles most of the infrastructure decisions you deal with in hosting web-accessible apps :-

  • Deployment and management are integrated into the platform.
  • Endpoints can be secured.
  • Sites can be scaled quickly to handle high traffic loads.
  • The built-in load balancing and traffic manager provide high availability.

App service offers automatic scaling and high availability features. You can scale up to 20 or 100 nodes (Vms underneath) depend on the pricing tier.

The pricing based on app service plan, a set of compute resources is created for that plan in that region. Each App Service plan defines: Regions, Number of VM instances, size of VMs and pricing tier. The pricing tier of an App Service plan determines what App Service features you get and how much you pay for the plan.

Three Pricing Tier categories are currently available :-

  • Shared Compute Dev/Test (Free and Shared) – The app is run on the same VM as other apps; scaling is not an option. This plan is good for development and testing environment. Ideal for less demanding workloads. Focused on providing shared infrastructure. Additional features include custom domains / SSL and manual scale.
  • Dedicated compute (Basic, Standard, Premium, Premium v2) Production – Apps are run on a dedicated VMs; apps in the same App Service plan shared the same compute resources. Ideal for more demanding workloads. Additional features include staging slots, daily backups, and a traffic manager.
  • Isolated - dedicated VMs run in dedicated vNETs. Ideal for workloads that require advanced networking and fine-grained scaling.

Within each category, there are different pricing tiers.

- With App Service, you can host most common web app styles, such as :

Web Apps - build and deploy web apps faster at scale. App Service supports different run-times and many programming languages hosting web apps like ASP.NET, ASP.NET Core, Java, Ruby, Node.js, PHP, or Python. You can choose either Windows or Linux as the host operating system.

API Apps – easily build and consume APIs. You can build REST-based web APIs by using your choice of language and framework.

Web App for containers - deploy and run containerized web apps.

WebJobs - WebJobs are often used to run background tasks as part of your application logic. You can use the WebJobs feature to run a program (.exe, Java, PHP, Python, or Node.js) or script (.cmd, .bat, PowerShell, or Bash) in the same context as a web app, API app, or mobile app. They can be scheduled or run by a trigger.

Mobile apps - Use the Mobile Apps feature of App Service to quickly build a back end for iOS and Android apps. With just a few actions in the Azure portal, you can:

  • Store mobile app data in a cloud-based SQL database.
  • Authenticate customers against common social providers, such as MSA, Google, Twitter, and Facebook.
  • Send push notifications.
  • Execute custom back-end logic in C# or Node.js.

2. Serverless Computing

Serverless is a cloud-native development model that allows developers to build and run applications without having to manage Infrastructures servers. With serverless computing you can build applications faster by eliminating the need to maintain virtual machines or containers.

With Azure, when talking about serverless computing, we need to refer to two azure services:

  • Azure Functions: Functions can execute code in almost any modern language.
  • Azure logic Apps: Logic apps are designed in a web-based designer and can execute logic triggered by Azure services without writing any code.

Serverless means that the infrastructure provision and management are invisible to the developer. There are still servers in serverless, but they are abstracted away from app development.

Servers are still running the code, everything is run behind the scene by azure cloud. We have three main pillars of serverless computing.

Serverless computing is a method of providing backend services on an as-used basis. Servers are still used, but a company that gets backend services from a serverless vendor is charged based on usage, not a fixed amount of bandwidth or number of servers.

- Abstraction of servers

Servers are completely abstracted from the underlying hosting environment. You don't need to configure or maintain the infrastructure you simply upload your code to azure and it runs with assuring High Availability and scalability.

Azure will automatically scale resources to maintain stable performance by allocation/deallocation of resources, You never explicitly reserve capacity.

- Event-Driven Scale

With serverless computing your code run based on trigger or events, for example it runs function when it receives an HTTP request, this process is good fit for workloads that respond to incoming events like notification message when you order something.

The platform automatically schedules the function to run and scales the number of compute instances based on the rate of incoming events. Either event invoked by users or event that will run everyday at a specific time.

A function contains both code and metadata about its triggers and bindings. Triggers define how a function is invoked. And Bindings provide a declarative way to connect to services from within the code. For example you order something the order function get triggered or called, within the function we have codes that binds with other functions or services that will allow us to generate a reply message to customer

Event include trigger by :

  • Timers For example, if a function needs to run every day at 10:00 AM UTC.
  • HTTP like API and webhook scenarios
  • Queues - with order processing

- Micro-billing (Pay by the run time)

You pay only for the duration your code runs, if your function is not active or no executions occur, then you are not charged. for example, if the code runs once a day for two minutes, they're charged for one execution and two minutes of computing time.

We can simply say :-

  • You only pay for the time the code runs.
  • No active function executions occur therefore there is not charge
  • For example, if the code runs once a day for two minutes, they're charged for one execution and two minutes of computing time.

- Azure Functions Service

Azure Functions is an event-driven, designed for Micro/Nano-services, it's a serverless compute option that doesn't require maintaining virtual machines or containers.

If you build an app using VMs or containers, those resources have to be running in order for your app to function. With Azure Functions, an event wakes the function, alleviating the need to keep resources provisioned when there are no events.

Using a virtual machine-based approach, you'd incur costs even when the virtual machine is idle. With functions, Azure runs your code when it's triggered and automatically deallocates resources when the function is finished. In this model, you're only charged for the CPU time used while your function runs.

Here some of trigger examples :-

  • Respond to data change
  • Run a task on schedule
  • Run a function as response to HTTP request

Azure Functions can execute code in almost any modern language. it commonly used when you need to perform work in response to an event.

Azure function is similar to app service, but the difference is in app service we were creating a full-fledged web services or web applications, but if we want to run a small piece of code, let’s say we have a function that adds two numbers a and b and return for us the result. We can use azure function to run this small piece of code.

We ask developers to prepare a small package and deploy our function up across multiple nodes (VMs underneath) and expose it as a web service. Here we host a small piece of code very effectively.

With Azure function service, auto-scaling feature is supported from 0 to 200 servers. It is one of the more scalable services in azure with it pricing consumption mode, where you can build scalable and cheap web applications.

two hosting/pricing model :-

  • Consumption based plan - pay exactly for what you use for example, 50 cents for one million execution.
  • Dedicated Plan using app services

Functions scale automatically based on demand, so they're a solid choice when demand is variable. For example, you might receive messages from an IoT solution that's used to monitor a fleet of delivery vehicles. You'll likely have more data arriving during business hours.

Azure Functions runs your code when it's triggered and automatically deallocates resources when the function is finished. In this model, you're only charged for the CPU time used while your function runs.

  • Stateless (the default) - They behave as if they're restarted every time they respond to an event.
  • Stateful (called "Durable Functions") - A context is passed through the function to track prior activity..
  • Stateful vs stateless

Benefits of Azure Functions :-

  • It's ideal when you're only concerned about the code running your service and not about the underlying platform or infrastructure.
  • We use it, when we want to perform work in response to an event often via a REST request, timer, or message from another Azure service.
  • Functions scale automatically based on demand, so they may be a good choice when demand is variable.

- Azure Logic Apps

Azure Logic apps are similar to azure functions, Both enable you to trigger logic based on an event, but with logic app you write nothing you don't need to write code. Azure functions execute code, while Azure logic apps executes workflows, using pre-built logic blocks that azure provide for us.

For example, a ticket arrives in ZenDesk, you could detect the intent of the message with cognitive services and then create an item in SharePoint to track the issue.

  • Also, you can add the customer to your Dynamics 365 CRM system if they aren't already in your database.
  • and Send a follow-up email to acknowledge their request.

Every Azure logic app workflow starts with a trigger, which fires when a specific event happens or when newly available data meets specific criteria. Many triggers include basic scheduling capabilities, so developers can specify how regularly their workloads will run. Each time the trigger fires, the Logic Apps engine creates a logic app instance that runs the actions in the workflow. These actions can also include data conversions and flow controls, such as conditional statements, switch statements, loops, and branching.

With Azure logic Apps you can :-

  • Schedule - basic scheduling capabilities
  • Automate ad orchestrate tasks
  • Business processes and workflows

When you need to integrate apps, data, systems and service across enterprises or organization. You create logic apps workflow using visual designer on azure portal or virtual studio.

Workflow like visualize, design, build, automate and deploy business processes as series of steps. Execute workflows designed to automate business scenarios and built from predefined logic blocks. The workflows are persisted as a JSON file with a known workflow schema.

Azure provides over 200 different connectors and processing blocks to interact with different services. You can also build custom connectors to interact. These resources include the most popular enterprise apps.

You can also build custom connectors and workflow steps if the service you need to interact with isn't covered. You then use the visual designer to link connectors and blocks together. You pass data through the workflow to do custom processing, often all without writing any code.

For example, when a customer opens the ticket to a service provider, ticket says “My TV Subscription is not working”. With Azure logic app, you could send an acknowledgment email that you have received the customer message.

3. Functions vs. Logic Apps

  • With Functions, you write code to complete each step.
  • With Logic Apps, you use a GUI to define the actions and how they relate to one another.
